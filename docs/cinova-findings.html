Section 1

Executive Summary

Cinova is a production-grade learning platform designed to improve first-time coding confidence while preserving technical rigor. This document consolidates the project's architecture, security posture, accessibility work, engineering decisions, and measurable outcomes for both technical reviewers and non-technical stakeholders.

The work described here focused on hardening an existing education platform for portfolio-grade quality: cleaner onboarding, stronger security controls, accessibility improvements, and a repeatable release workflow — all delivered through small, check-validated batches without a full architectural rewrite.

Section 2

Product Overview

Cinova is a multi-track learning platform covering HTML, CSS, JavaScript, React, and Python. Each lesson follows a repeatable six-step model:

Hook — engage with a real problem
Do — hands-on activity first
Understand — concept explanation after practice
Build — apply the concept in code
Challenge — extend without scaffolding
Summary — consolidate and reflect

Key Features

  • Guided lessons with live coding and immediate feedback loops
  • Monaco-based code editor and preview workflows
  • AI tutor and AI-generated practice cards (server-mediated)
  • Course navigation, bookmarks, glossary, and search
  • Spaced repetition queue built from missed concepts
  • Gamification: XP, streaks, badges, and daily momentum goals
  • Public learner profile pages with opt-in privacy controls
  • Admin dashboard and lesson content tooling
  • PWA support and offline-oriented behavior

Design Philosophy

Most tutorial apps optimize for content volume. Cinova optimizes for learning momentum and production behavior. This means clearer "what to do next" guidance, real state persistence, secure AI integration, and consistent empty/loading/error states across all paths.

Section 3

Architecture & Technical Design

Browser (React 18 + Vite) │ ├── Route shell and feature panels │ ├── Supabase JS client │ ├── Auth session management │ └── Postgres reads/writes (RLS enforced) │ ├── Netlify Functions │ ├── /.netlify/functions/ai │ ├── /.netlify/functions/practice-generate │ └── /.netlify/functions/streak-reminder │ └── OpenAI Responses API (server-side only)

Design Invariants

  1. No secrets in the browser bundle. API keys never reach the client.
  2. RLS is the true authorization boundary. The UI is a convenience layer, not a security layer.
  3. Paid AI calls are server-mediated and rate-limited. All OpenAI traffic goes through authenticated Netlify Functions.
  4. Security-critical dependencies fail closed. If the rate limiter is unreachable, the request is rejected rather than forwarded.

Styling Architecture

The UI uses tokenized CSS variables and shared component patterns. Style imports flow in a strict order through src/styles/App.css:

  1. tokens.css — design tokens (colors, spacing, radii, typography)
  2. base.css — resets and base element styles
  3. animations.css — shared motion primitives
  4. Feature modules: auth.css, sidebar.css, lessons.css, panels.css, landing.css, responsive.css
  5. Domain modules: shell-layout.css, learning-experience.css, feature-surfaces.css, platform-quality.css

Section 5

Security Findings

Threat Model Summary

AssetThreatsMitigations
Learner accounts Account takeover via XSS, session theft, OAuth redirect hijack HTML-escaping in all dangerouslySetInnerHTML sinks, strict CSP, Supabase URL allowlist for OAuth
Learner data (progress, notes) Cross-tenant read/write Row-Level Security on every Supabase table (auth.uid() = user_id)
OpenAI API key Exfiltration, free LLM abuse Key is server-only; Netlify Function gateway requires a valid session, applies per-user rate limiting, caps payload size, prepends server-side guardrail prompt
Admin surface Privilege escalation Direct UPDATE to is_admin blocked by Postgres trigger; promotions must go through set_user_admin() RPC, which refuses self-edits and writes to admin_audit_log
Code playground Sandbox escape to parent origin <iframe sandbox="allow-scripts"> without allow-same-origin
Build pipeline Malicious dependencies, supply-chain compromise Committed lockfile, npm audit in CI, gitleaks secret scanning, Dependabot

Security Headers (netlify.toml)

  • Content-Security-Policy — default-src 'self' with explicit allowlist; frame-ancestors 'none'
  • Strict-Transport-Security — 2-year HSTS with includeSubDomains and preload
  • X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: strict-origin-when-cross-origin
  • Permissions-Policy disables camera, microphone, and geolocation
  • Cross-Origin-Opener-Policy: same-origin and Cross-Origin-Resource-Policy: same-origin

XSS Fixes Applied

  • renderMarkdown (src/utils/markdown.jsx) now HTML-escapes before applying markdown substitution
  • SearchPanel.highlight() escapes both source text and query before regex replacement
  • DOMPurify pinned via package.json overrides

AI Proxy Controls

  • Requires a valid Supabase bearer token verified against /auth/v1/user
  • Persistent per-user rate limit in Postgres via public.consume_ai_quota() SECURITY DEFINER RPC (10 requests / 60 seconds)
  • Strict payload caps: system prompt ≤ 2000 chars, ≤ 20 messages, ≤ 4000 chars/message, ≤ 12000 chars total, ≤ 1024 output tokens
  • Mandatory server-side guardrail prefix prevents repurposing as a general-purpose LLM
  • Fails closed — if the rate-limit RPC is unreachable, the request is rejected (503) rather than forwarded

Section 6

Accessibility Hardening

Accessibility improvements were built into shared primitives so they apply consistently across all surfaces rather than as one-off patches.

Semantics

Semantic landmarks and structured heading hierarchy throughout. Stronger dialog semantics with aria-labelledby and aria-describedby.

Keyboard Navigation

Keyboard-navigable dialogs and overlays with proper focus management. Visible focus states across all interactive controls.

ARIA

Meaningful labels and ARIA wiring for high-traffic interactions. Cleaner interaction semantics in quiz, search, and sidebar flows.

Motion

Reduced-motion support across all animations. The scroll-driven landing hero respects prefers-reduced-motion with no additional dependencies.

Section 7

Key Technical Tradeoffs

Tradeoff 1 · Polish Speed vs Architectural Churn

Decision: Optimize within current architecture instead of introducing a new state or design framework.

Why: Reduced risk, preserved momentum, and kept changes reviewable in small batches. Tokenized CSS and shared UI primitives provided consistency without a heavy design-system dependency.

Tradeoff 2 · Richer UX Semantics vs Test Stability

Decision: Align ARIA semantics with test expectations and accessibility best practices instead of forcing custom patterns.

Why: Prevented regressions while improving screen reader and keyboard behavior simultaneously.

Tradeoff 3 · Strict Release Confidence vs Local Iteration Speed

Decision: Use simple command gates (check, check:ci) to keep both fast feedback and CI parity.

Why: Stronger release confidence without a complex pipeline redesign.

Tradeoff 4 · AI Usefulness vs Safety and Cost Control

Decision: Keep all AI calls server-mediated with strict payload controls and per-user limits enforced in the database, not in the function.

Why: A function-level-only limit could be bypassed by concurrent requests. Moving the limit to a Postgres SECURITY DEFINER RPC that reads auth.uid() server-side closes that window.

Tradeoff 5 · Fast Initial Load vs Feature Depth

Decision: Lazy-load Monaco editor, admin views, and styleguide. Self-host fonts with Latin subset declarations. Add a bundle budget gate to CI.

Why: First-run learners should not pay the editor or admin cost upfront. The budget gate prevents silent regressions as features are added.

Section 9

Performance Strategy

  • Domain-aware chunking in vite.config.js keeps first paint focused on the learning shell.
  • Monaco editor is lazy-loaded so first-run users do not pay the ~1 MB editor cost upfront.
  • Admin views are lazy-loaded and route-gated, adding no cost to the learner path.
  • Bundle budget checks (npm run check:bundle) are enforced in CI to prevent silent regressions — stricter on eager chunks, with explicit allowances for Monaco-specific budgets.
  • Self-hosted fonts: Inter (variable, Latin + Latin-ext subset), Poppins and Space Mono (Latin-scoped) — eliminates runtime Google Fonts requests and reduces first-load font payload.
  • Lighthouse CI runs score thresholds and budget gates in CI to prevent performance regressions from reaching production.

Section 12

Lessons Learned

  • UX polish is mostly consistency, not flashy effects. Standardizing spacing, radii, and border rhythm produced a premium feel far more efficiently than adding new visual effects.
  • Accessibility is easier to sustain when built into shared primitives. Once focus management and ARIA patterns lived in shared components, every new surface inherited them for free.
  • Small, check-validated batches produce safer momentum than rewrites. Each batch could be reviewed, reverted, and understood independently.
  • Security controls belong at the backend boundary. Rate limits, authorization, and payload validation in the Netlify Function and Postgres layer — not in client state — meant the client could be fully untrusted without sacrificing safety.
  • Recruiter-facing documentation should explain both product value and engineering tradeoffs. Showing what was built and why specific tradeoffs were made is more compelling than a feature list alone.
